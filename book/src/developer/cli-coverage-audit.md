- visible command help and per-subcommand help
- version output
- setup/openapi/cache utility paths
- selected argument-validation and exit-code behavior
- daemon lifecycle subprocess coverage
- server-backed subprocess coverage
- explicit `transcribe --server` local-audio fallback coverage, including the

- declarative command-family matrix coverage for `translate`, `coref`,
  `utseg`, `compare`, `opensmile`, and `avqi`
- common I/O mode coverage for `--file-list`, `--before`, output-dir
  creation, and representative invalid-path contracts
- explicit `--server` capability-rejection contracts for unsupported commands
- release-facing command manifest coverage
- hidden batchalign2 compatibility acceptance coverage
- hidden batchalign2 alias equivalence coverage for `align`, `transcribe`, and
  `benchmark`, including custom-engine and diarization precedence on the current
  typed-option builder path

The important improvement from this pass is that the coverage is no longer just
ad hoc spot tests. The CLI surface itself is now documented and asserted.

## Highest-value remaining gaps

The next work should focus on the gaps that remain after manifest, harness, and
alias-equivalence landings:

1. **Backend-dependent runtime coverage**
   - The command matrix now protects parse, dispatch-routing, and subprocess
     contracts for the missing families.
   - Real success-path execution for infer/media commands still depends on
     infer-capable workers or media-analysis backends beyond the lightweight
     harnesses used for most of the matrix.
   - We now protect one full live `align --server` FA success path and one live
     local-daemon `.mp4` transcribe path that proves injected media-cache
     population, but the broader backend matrix is still incomplete.

2. **Option-family conflict and precedence coverage**
   - `--wor` vs `--nowor`
   - remaining bool-pair interactions beyond the now-covered
     `--merge-abbrev`/`--no-merge-abbrev` matrix and the newly covered
     batchalign2 alias precedence checks
   - broader precedence outside the hidden BA2 alias layer, especially for
     engine overrides and remaining bool-pair families

3. **Python-facing CLI entry coverage**
   - The Rust binary is now much better covered than the Python entry path.

## Recommended next sequence

The most productive next sequence is:

1. extend `ba-runtime-environment-core` so stateful CLI tests rely on explicit
   runtime ownership instead of ambient machine state
2. add worker/backend coverage where CLI behavior depends on backend routing
3. widen option-family conflict coverage beyond the now-protected alias cases
4. add Python-entry coverage so the published console-script path stays in sync

This ordering keeps using the new manifest and harness layers as protection
while the remaining runtime seams are extracted.

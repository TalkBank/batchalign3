# Tier Parse Leniency — Design Issue

**Date:** 2026-02-14
**Status:** Documented for future study; immediate fix is tier-disposition based.

## Problem

CHAT files in the wild contain dependent tiers (%wor, %mor, %gra, %pho) with
varying levels of data quality. Some were generated by batchalign, some by CLAN,
some by hand. When batchalign needs to process a file (e.g., re-align), errors
in tiers it doesn't need should not block processing.

### Examples of real-world %wor problems

1. **Events in %wor**: `&=laughs` appears in %wor tier (not generated by
   batchalign — pre-existing data quality error)
2. **Full main tier syntax in %wor**: `<on a> [/]`, `[: tipping] [* p:w]`
   appear in %wor (generated by CLAN, valid but not handled by flat parser)
3. **Word count mismatch**: %wor word count doesn't match main tier (stale tier)

## Current Parsing Modes

- **Strict** (`parse_strict_pure`): Rejects on ANY error. Used by morphotag
  (which reads its own output — must be clean).
- **Lenient** (`parse_lenient_pure`): Error recovery, marks tainted tiers.
  Used by align, translate.

## Proposed: Tier Disposition Policy

Each command declares per-tier disposition:

| Disposition | Behavior on parse error | Use case |
|-------------|------------------------|----------|
| **READ** | Fail the file | Main tier, headers (always required) |
| **WRITE** | Drop the tier silently | Tiers being regenerated (e.g., %wor for align) |
| **Pass-through** | Preserve raw text, skip parse | Tiers not used by this command |

### Per-command dispositions

| Tier | align | morphotag | translate | utseg |
|------|-------|-----------|-----------|-------|
| Main tier | READ | READ | READ | READ |
| Headers | READ | READ | READ | READ |
| %wor | WRITE | pass-through | pass-through | pass-through |
| %mor | pass-through | WRITE | pass-through | pass-through |
| %gra | pass-through | WRITE | pass-through | pass-through |
| %pho | pass-through | pass-through | pass-through | pass-through |

## Future: ParseHealth (per-utterance, per-tier)

For analysis tools that need to READ dependent tiers, a richer model is needed:

```
utterance.tier_health("wor") -> Ok | ParseFailed(error) | Missing | Stale
```

- **Ok**: Tier parsed and is semantically consistent with main tier
- **ParseFailed**: Tier had syntax errors (with error details)
- **Missing**: Tier doesn't exist for this utterance
- **Stale**: Tier parsed but word count mismatches main tier

Analysis tools would query for utterances with healthy tiers and report
coverage statistics ("analyzed 97/100 utterances with valid %wor").

This prevents the dangerous case: silently consuming malformed tier data.
The protection is making health status impossible to ignore — metadata always
attached to the result, not a flag you opt into.

## Immediate Fix (2026-02-14)

For now: revert %wor grammar to use `tier_body` (same as main tier). This
handles all real-world %wor variants. The semantic %wor parser extracts
words + timing from whatever structure the grammar produces.

The flat `wor_tier_body` assumption was wrong — real CLAN-produced %wor
contains groups, retraces, replacements, and annotations.
